# Watch platform — v0.2

## Summary

v0.1 stood the Watch estate up on AWS: a live vertical slice (ECS Fargate behind an ALB,
CodeDeploy blue/green, AppConfig flags, Step Functions escalation, intake, a status page,
promote-by-digest) reachable at `watch.davestanton.com`.

**v0.2 turns that slice into a lifecycle you can run with confidence.** The estate can now be
stood up and torn down deterministically with a **single human approval each way**; real code
is promoted through a **three-gate chain — SAST (Sonar) → DAST (ZAP) → functional smoke
(Playwright)** — before it can reach prod; the app is **HTTPS-hardened** and
**backend-agnostic-observable**; **staging is prod-identical and browser/DAST-testable**; and
**cost is guardrailed per plane** (prod / staging / platform). A push to `main` deploys itself.

The through-line is **determinism and gating**: every lifecycle operation is a `make` target
backed by a re-runnable script, the two consequential levers (create, destroy) are the only
things that prompt a human, and nothing reaches prod that hasn't passed the same gates locally
and in CI.

Delivered: **#18, #24, #28, #30, #31, #32, #33, #34, #35, #37**, plus the deterministic tooling,
the audit-gate policy, and the E2E smoke suite (23 commits since `v0.1.0`).

## At a glance

| Area | v0.1 | v0.2 |
|---|---|---|
| Lifecycle | manual `terragrunt` per stack | `make live` / `make teardown` — one approval each, DAG-parallel |
| Release | manual `start-pipeline-execution` | push to `main` auto-deploys (GitHub Actions → OIDC → CodePipeline) |
| Gates | Sonar (SAST) at build | **SAST → DAST → functional smoke**, each blocking prod |
| Staging | HTTP-only on the raw ALB | `watch-stg` / `status-stg` HTTPS — browser- + DAST-testable |
| TLS/DNS | prod only, combined stack (latent bootstrap cycle) | both envs, **cert split from records** — deterministic from scratch |
| Telemetry | off in AWS | **Alloy sidecar**, app names no backend (ADR-016) |
| HTTP hardening | HTTPS-only | + HSTS, CSP, X-Frame-Options, Permissions-Policy, nosniff, referrer |
| Cost | one $10/day account budget | prod / staging / **platform** daily budgets + `env` cost tag |
| Human touch | ad-hoc CLI | **one approval to create, one to destroy**; prod-promote approval |

## Capabilities: how they were built and tested

### 1. Deterministic lifecycle & the audit gate

**Built.** Every lifecycle operation is now a checked-in script wrapped by a `make` target:

- `scripts/create.sh` — `terragrunt run --all apply`; the DAG orders and parallelizes
  `ecr → connection → ci-trigger → per-env (network → data/config → cert → app → escalation/
  intake/frontend/observability → dns) → pipeline`. Self-heals a missing `:bootstrap` image and
  auto-stops the V2 pipeline's create-triggered run.
- `scripts/teardown.sh` — dependents-first, `--parallel` across envs, drops the `watch`/`status`
  CNAMEs (keeping the certs), keeps the ~$0 foundation.
- `scripts/db.sh` (migrate/seed as one-off Fargate tasks), `scripts/deploy-frontend.sh`
  (SPA → S3 + invalidation), `scripts/deploy.sh` (promote latest `main` to the approval gate),
  `scripts/sweep.sh` (read-only billable-orphan gate).
- `make live` = `create` (both envs) → `migrate` + `seed` (staging **and** prod) →
  `deploy-frontend` (both) → `deploy` (promote latest `main`). One command stands up the
  platform + both app envs.

The **audit model** lives in `.claude/settings.json`: everything the machine needs is
pre-allowed; the **only** actions that prompt a human are the **create** and **destroy** levers
(and the raw paths they wrap, so they can't be reached un-prompted). `deny` > `ask` > `allow`,
so the broad allow can't slip an apply/destroy past the gate.

**Tested.** A full **teardown → `make live` from scratch** was exercised end-to-end: 18 stacks +
pipeline destroyed and swept clean, then recreated in **one apply pass** — no `CNAMEAlreadyExists`,
no `DuplicateListener`, no bootstrap cycle. The parallel teardown was verified to keep the
foundation (both certs, ECR, connection, ci-trigger, state) and drop only the CNAMEs; the sweep
confirmed zero billable orphans. Two human approvals bracketed the whole cycle.

### 2. Release automation (#24, #33, #28)

**Built.** A push to `watch` `main` now deploys itself:

- The native CodeConnections push trigger was configured correctly but **never delivered
  events** (a GitHub-App-side gap invisible to the AWS CLI). So the trigger is a **GitHub Actions
  workflow** (`trigger-pipeline.yml`) that assumes a least-privilege OIDC role
  (`watch-ci-trigger`: trust `gotoplanb/watch` any ref, permission = `StartPipelineExecution` on
  the one pipeline) and starts the pipeline. Trigger-only — CodePipeline still builds/deploys
  (ADR-004). Swappable to release tags with a one-line `on:` change.
- **#33**: the GitHub connection was moved to its **own persistent stack** (`modules/connection`,
  `prevent_destroy`), so teardown/recreate keeps it `AVAILABLE` — authorize once, never again.
- **#28**: `make pipeline` re-applies the pipeline to repoint at fresh staging ARNs after a
  staging-only recreate.

**Tested.** Validated end-to-end: a push → the GHA workflow succeeded → a new `StartPipelineExecution`
appeared → Build → DeployStaging → prod approval. Confirmed the promote-by-digest identity holds
(same `sha256` staging→prod). After a teardown/recreate, the connection stayed `AVAILABLE` (no
re-auth) and the freshly-created pipeline carried the full gate chain.

### 3. DNS/TLS for both envs + the cert split (#34, #35)

**Built.** Staging gained real hostnames (`watch-stg` / `status-stg`.davestanton.com) so it can be
driven over HTTPS by a browser, Playwright, and DAST. Doing so surfaced a latent **bootstrap
cycle** — the app needs the ACM cert, but a combined dns+cert stack depends on the app for the
CNAME targets. The fix was to **split the cert into its own stack** (`modules/acm-cert`, applies
first, no app dependency) from the **records** (`modules/dns-records`), with the app/frontend
taking the cert ARN as an input. Applied to **both** envs (ADR-019 parity); prod's live cert was
**migrated in place** (`import` → `state rm` → config switch, cert never recreated).

**Tested.** Staging: `watch-stg` served **200 over HTTPS**, `status-stg` rendered live posture in
the browser (verified in Playwright), login worked. Prod migration: the ALB `:443` cert ARN was
**unchanged** and `watch`/`status`.davestanton.com **stayed 200 throughout** (`prod/app` +
`prod/frontend` plans were clean no-ops). The from-scratch `make live` then proved cert-first
ordering works for both envs with no manual bootstrap.

### 4. Backend-agnostic observability (#18, ADR-016)

**Built.** The app exports OTLP to a **local Alloy sidecar** (`essential=false`) and names no
telemetry backend. `OTEL_EXPORTER_OTLP_ENDPOINT` → `localhost`;
`OTEL_RESOURCE_ATTRIBUTES=deployment.environment=<env>,service.version=<git sha>` (the pipeline
injects the SHA); the remote-endpoint var was dropped. The sidecar batches and forwards to a
per-env gateway (or a debug sink until one is wired).

**Tested.** On staging, all three app containers ran and the sidecar received the app's **traces,
metrics, and logs**. Crucially, when a config bug crashed the sidecar, **the app stayed RUNNING**
— `essential=false` kept telemetry off the critical path exactly as ADR-016 requires.

### 5. HTTP hardening (#30, #37)

**Built.** HSTS on both status pages (CloudFront response-headers policy) and the app
(`SECURE_HSTS_SECONDS` behind the ALB). The DAST scan then flagged missing app-response headers,
which #37 added: `X-Frame-Options: DENY` + nosniff + referrer-policy via Django, and CSP +
Permissions-Policy via a small middleware (CSP permits the `/ui` CDN deps — Tailwind Play,
Alpine/HTMX — and locks the rest).

**Tested.** Verified live in prod: `watch.davestanton.com` serves the full header set
(`strict-transport-security`, `content-security-policy`, `x-frame-options`, `permissions-policy`,
`x-content-type-options`, `referrer-policy`). The DAST re-run measured the improvement:
**WARN 12 → 10, PASS 55 → 57**, closing the loop DAST itself opened.

### 6. The quality-gate chain: SAST → DAST → functional smoke (#32 + E2E)

**Built.** The pipeline is now `Source → Build → DeployStaging → DAST → Smoke → ApproveProd →
DeployProd`:

- **DAST** — a CodeBuild stage runs an OWASP ZAP baseline against `watch-stg` (public, no VPC),
  uploads the report to the artifact bucket. Runs with `-I` so it's non-breaking to start (only
  FAIL-promoted alerts gate; tighten via a `.zap` rules config).
- **Smoke** — a CodeBuild stage (Playwright image) runs the app repo's `e2e/` suite against
  `watch-stg`: **health → status (Postgres/Valkey checks + SPA render) → login `t1a` → intake
  create → escalate → poll to T2**. So RDS (read+write), Valkey (session), Step Functions +
  commit Lambda (escalate), and the status SPA are each exercised post-deploy. The intake secret
  is injected from SSM. Failure blocks the prod promotion.
- **Local mirror** — `make e2e` runs the same suite against the `make dev` loop, and the
  pre-commit hook runs coverage + Sonar + the e2e smoke (conditional on the local app being up),
  so local gates mirror CI as closely as practical. Node was added to the toolchain for this.

**Tested.** The suite was validated by running the actual spec against `watch-stg` (passes in
~15s) — and running it via node caught two real bugs curl/manual checks couldn't: seeded
incidents **auto-escalate off T1** on their SLA timeout (so the test tracks its *own* incident),
and the escalate **no-ops if clicked before the Step Functions task token registers** (so it
retries while at T1, guarded against over-escalating). The gate was then proven **in both
directions in CI**: a broken spec produced `Smoke = Failed` and **blocked the pipeline before
ApproveProd**; the fixed spec proceeds.

### 7. Cost guardrails per plane (#31)

**Built.** Three daily budgets — `watch-prod-daily`, `watch-staging-daily`, and
`watch-platform-daily` (the shared plane: ECR, connection, ci-trigger, and the future Watchtower
slice + Sonar) — each scoped by the `env` cost-allocation tag, plus a tuned $20 account cap. The
`env` tag was activated for cost allocation and the platform stacks tagged `env=platform`.

**Tested.** All four budgets applied; the `env` cost-allocation tag activated. (Cost-allocation
tags carry a ~24h activation lag and count spend forward-only, noted for validation.)

## Operational reference

```
make live                 # ONE approval: platform + BOTH app envs, then promote latest main
make teardown             # ONE approval: destroy both envs + pipeline (keeps certs/ECR/connection)
make deploy               # promote latest main through the pipeline to the prod-approval gate
make create[-staging|-prod]   make migrate/seed ENV=…   make deploy-frontend ENV=…
make pipeline             # re-apply the pipeline (repoint at current env ARNs)
make sweep                # read-only billable-orphan check (nonzero exit = something lingers)
# watch repo: make e2e    # Playwright smoke vs local (or override E2E_BASE/E2E_STATUS/E2E_SECRET)
```

Two human touchpoints in the release path: the **`make live` approval** (stand up the estate)
and the pipeline's **prod-promote approval** (ship real code to prod). Everything between is
gated and automated.
